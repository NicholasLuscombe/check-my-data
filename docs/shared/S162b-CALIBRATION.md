\# S162b calibration — per-test prose anchors Chat-owned design artefact. Locks the `Finding.location` and `Finding.evidenceLines` register for the §4 prompt body composers (S162b implementation session). Each entry pairs a real result object harvested from the engine with a locked target prose output — the goal is that a Code composer for the same test, given the same result object, produces the locked output (or a defensible variant). ## Source artefacts - Engine harvest: [`docs/test-artefacts/S162b-anchor-harvest.md`](../test-artefacts/S162b-anchor-harvest.md) (generated by `test/diag-s162b-anchor-harvest.mjs`). - Phase A behaviour: `handoffModel.js` `locationOf(r)` returns `r.testLocalisation`; `evidenceOf(r)` returns `[r.interpretation, "primaryP = …"]`. - Locked S160 design pass register: the three pre-S162b fragments (LOESS / Benford / IRC on DS08) embedded in SESSION161-SUMMARY.md. Calibration here refreshes against current engine output and extends to four more anchors. ## Phase A reality check Of the seven anchors, only LOESS and IRC populate `r.interpretation`. The other five (Benford, Selective Noise, Mahalanobis, DupDet, Baseline Balance) return no interpretation string at all, so Phase A's `evidenceLines = [r.interpretation, primaryP]` currently emits `[undefined, "primaryP = …"]`. S162b is a correctness fix on the §4 emit, not just an enrichment pass. --- ## Register decisions (cross-anchor principles) Locked during authoring; the composer registry follows these rules. ### Location Single line, no trailing period (locations aren't sentences). One of: - `Global` — tests with no spatial localisation (Benford, Carlisle, global statistics). - `Row N` or `Rows N, M, P` — single rows or small sets (≤5 rows). - `Rows N, M, P, … (K of TOTAL total)` — large sets with truncation; cap at 5 inline, summary tail. - `Rows A–B` — contiguous ranges; multiple ranges comma-separated. - `Pair X–Y` or `Pairs X–Y, Z–W` — replicate-pair anchors. - `Column N` or `Columns N, M` — column-level anchors. - `Changepoint at row N` — anchor-point tests (LOESS CUSUM). - `K block copies` / `K rows in 3 duplicate groups` — structural summaries when individual indices are too numerous to be informative. Locations may compose multiple anchor types separated by semicolons (e.g. `Changepoint at row 32; smoother windows in rows 25–62`). ### Evidence lines 1–3 lines per finding. Each line is a complete sentence ending in a period. - **Line 1** (always): headline statistic + classification verdict. The statistic that drove the flag, with its p-value or other significance indicator, plus the classification register the test uses ("Nonconforming", "smoother", "suspicious", "too balanced"). - **Line 2** (when richer detail exists): per-element detail — top-K windows, digit distribution, per-pair statistics, per-column variances. The "where the action is" line. - **Line 3** (when meaningfully informative): supporting context — mechanism summary, region comparison, qualifier ("one of three pairs flagged"), or N count. Prefer 1–2 dense lines over 3 sparse lines. Three lines reserved for tests with genuinely rich per-element detail (Benford's full 9-digit distribution, LOESS's region-comparison-plus-windows). ### Statistical precision - p-values: 4 decimal places for small values (0.0002, 0.0006). `< 0.0001` at floor (when result reports `"<0.0001"` or `0`). Drop trailing zeros below 4dp (0.005 not 0.0050). - Ratios: 2dp + `×` suffix (`7.17×`, `2.50×`). For sub-unity ratios prefer the inverse with direction word (`0.42× (col 3 quieter)`). - Test statistics: preserve the engine's own precision verbatim — do not trim. `D = 0.7512` not `D = 0.751`; `χ² = 30.83` (engine emits 3 sf); `cusumStat = 2.169` (engine emits 4 sf). Report df where canonical (`χ² = 30.83 on 8 df`). - r-values: 4dp matching the IRC engine's own precision (`r = 0.9906`). - Counts: integers (`N = 195`, `5 of 60 features`). - Percentages: 1dp (`19.0%`, `4.6%`). ### Statistical idiom — reader-friendly, not engine field names p-values and test statistics are named in reader-comprehensible idiom, not engine field names. The §4 prompt body is consumed by an LLM and a domain reader, not by code that needs to grep field names. Triangulation locked this as a substantive register choice across all three models. - `CUSUM p = 0.0002` — not `cusumP` - `scan p = 0.002` — not `scanP` - `Bartlett p = 0.0006` — not `pBartlett`. When the test statistic itself appears alongside, drop the redundant "Bartlett" prefix on the p: `Bartlett χ² = 14.88 on 2 df (p = 0.0006)`. - `binomial p = 0.0017` — not `binomP` - `KS p < 0.0001` — not `ksP`; spell out `Kolmogorov–Smirnov D = 0.7512` on first mention within a finding, `KS D` acceptable on second mention if any. - `MAD p < 0.0001` — not `pMAD` - `BH-adjusted p = 0.0006` — when the BH-FDR sub-unit promotion produces the per-pair / per-column p-value; this is the IRC and per-column SelNoise idiom. Abbreviations the audience knows (BH-FDR, ANOVA, KS, χ², df, SD, r) stay short. Abbreviations that are engine internals or non-standard (LOO, RSC, IRC outside section headers) spell out on first mention. ### No paraphrased examples Composers must cite engine field values verbatim or aggregate statistics computed from them — never paraphrased "representative examples." If the line says `per-feature p-values include 0.998, 0.999, 1.000`, those three numbers must appear in the engine's output array; if they don't, the line is a hallucination even if numerically close to engine values. The composer's source-of-truth is the result object; nothing else. Aggregate statements are fine: `48 of 60 features have p ≥ 0.9`, `digit 4 deviates by +7.2pp`, `mean inter-replicate r = 0.9771` — these all derive from the array via a defined operation (count, deviation, mean). Hand-picked illustrative values are not. ### Composer-layer truncation Engine fields may emit uncapped arrays (S162a's `flaggedRowIndices`, DupDet's 24-element details). Composers cap inline: - Index lists: 5 inline, then `… (K of TOTAL total)` summary. - Window lists: 3 inline (matches locked LOESS register). - Per-column / per-pair lists: full enumeration up to 6–8; beyond that, condense to flagged-only. - Digit distributions: full 9-digit listing acceptable (Benford 1st only); collapse middle range when overflow (`6–9: 5.6–7.7%`). The composer makes the truncation decision; the renderer pastes the resulting string verbatim. No truncation logic in promptBodyRenderer. ### Voice - Sentence-case ("Smoother windows at…", not "SMOOTHER WINDOWS AT…" or "smoother windows at…" mid-sentence). - Active where possible. Passive when the statistical convention demands it ("rows flagged as suspicious"). - Mechanism language (smoother / quieter / nonconforming / too balanced / suspicious) is the test's own classification register — match the test's existing vocabulary; do not paraphrase to a generic synonym. - Numerals for all numeric quantities, including `1`, `2`, `3` (`1 pair of three`, not `one pair of three`). - En-dashes for ranges (`rows 37–56`, `1–3 evidence lines`); en-dashes also for pair labels (`pair 2–3`). - **No test-name restatement.** The test name appears as the §4 section header above each finding ("Benford's Law (First Digit)"); evidenceLines do not restate it ("Benford's Law classified the dataset as HIGH…"). Composers open line 1 with the headline statistic, not with a meta-sentence about which test ran. --- ## Anchor 1 — LOESS Residual Analysis (DS08) **Test name (canonical):** `LOESS Residual Analysis` **Fixture:** `08-elisa-fabricated.csv` (65 × 3, ELISA) **Flag:** HIGH; primaryP = 0.0002 **Fields consumed:** - `changepointRow` (32), `changepointDirection` ("decreases"), `cusumP` (0.0002) - `details` (top-K windows + changepoint, already pre-formatted: `rows`, `ratio`, `direction`) - `regionComparison` (pre/post changepoint noise vs expected: `region`, `rows`, `observedNoise`, `expectedNoise`, `ratio`, `finding`) - `scanP` (0.002), `bestVarRatio`, `bestWindowRows` - `pairResults` (per-pair scan/cusum p-values) — informational, not surfaced in DS08 prose since `pairPromoted=false` **Locked location:** ``` Changepoint at row 32; 4 smoother windows in rows 25–62 ``` **Locked evidence lines:** ``` Changepoint at row 32 (noise decreases after this point); smoother windows at rows 37–56 (7.17× variance ratio), 43–62 (6.34×), 25–44 (4.87×).
